## 1. 架构概述

### 1.1 设计目标

#### 核心目标

- **持久化知识积累**: 不是简单的 RAG 检索，而是构建持续进化的知识体系
- **增量式更新**: 每次添加新文档时自动提取、整合、更新
- **结构化关联**: 文档间自动建立交叉引用，形成知识网络
- **合成与演进**: 从原始文档到综合论述，层层提炼

#### 非功能性目标

- **可扩展性**: 支持从 10 篇到 10,000 篇文档的增长
- **可维护性**: LLM 自动维护，人工干预最小化
- **可追溯性**: 所有信息有来源，所有变更有记录
- **可用性**: 查询响应时间 < 30 秒

### 1.2 架构原则

#### 单一职责原则

每个目录、文件、页面都有明确的职责：

```
raw-docs/     → 存储原始文档（只读）
wiki/         → 存储衍生知识（LLM 维护）
indexes/      → 提供检索入口（自动更新）
logs/         → 记录操作历史（追加写入）
templates/    → 定义页面结构（稳定不变）
schema/       → 配置系统行为（定期更新）
```

#### 关注点分离

- **数据层**: raw-docs/, wiki/, indexes/
- **逻辑层**: schema/配置 + LLM 推理
- **展示层**: Markdown 渲染（Obsidian 等）

#### 不可变性

- ✅ **raw-docs/** 中的文档不可修改（只能新增版本）
- ✅ **logs/** 中的记录不可修改（只能追加）
- ✅ **wiki/** 中的页面可以更新（但保留版本历史）

### 2.2 各层职责

#### 展示层

**职责**: 提供用户界面

**技术选项**:
- Obsidian（推荐）- 本地 Markdown 编辑器
- VS Code - 代码编辑器
- Web UI - 自定义前端（可选）

**要求**:
- 支持 Markdown 渲染
- 支持内部链接
- 支持图谱视图（可选）

#### 应用层

**职责**: 智能处理

**核心组件**: LLM Agent

**功能模块**:

```
LLM Agent
├── 文档处理模块
│   ├── 解析器（Markdown 解析）
│   ├── 提取器（实体/概念提取）
```

## 3. 数据流

### 3.1 文档摄入流程

```
用户添加文档
    ↓
[raw-docs/requirements/20260417-feature-spec.md]
    ↓
LLM 检测新文档
    ↓
读取文档内容
    ↓
提取信息
├── 核心摘要
├── 实体列表
├── 概念列表
├── 关键决策
└── 待办事项
    ↓
创建/更新 Wiki 页面
├── wiki/summaries/Summary-feature-spec.md
├── wiki/entities/Entity-FeatureModule.md
└── wiki/concepts/Concept-FeatureFlag.md
    ↓
更新索引
├── indexes/entity-index.md
├── indexes/concept-index.md
└── indexes/document-index.md
    ↓
记录日志
├── logs/ingestion-log.md
└── logs/change-log.md
    ↓
通知用户
```

### 3.2 查询响应流程

```
用户提问
    ↓
"认证模块是如何设计的？"
    ↓
LLM 分析查询意图
    ↓
检索 Wiki
├── wiki/entities/Entity-AuthenticationModule.md
├── wiki/concepts/Concept-CQRS.md
└── wiki/synthesis/Synthesis-Auth-Patterns.md
    ↓
评估信息充分性
    ↓
    ├─ 充分 → 综合回答
    └─ 不足 → 检索原始文档
        ↓
        [raw-docs/design/auth-design.md]
        ↓
        综合回答 + 知识缺口提示
    ↓
记录查询
    ↓
logs/query-log.md
```

### 3.3 Wiki 维护流程

```
定期触发（每周）
    ↓
检查实体页面
├── 孤立实体
├── 过时实体
└── 缺失实体
    ↓
检查概念页面
├── 孤立概念
├── 过时概念
└── 缺失概念
    ↓
检查交叉引用
├── 验证链接
└── 修复断开链接
    ↓
识别综合机会
└── 创建 synthesis 建议
    ↓
生成维护报告
    ↓
提交用户审查
```

#### 目录分片

当某个目录的文件数量超过阈值时自动分片：

```
阈值：50 个文件/目录

entities/ (52 个文件) → 触发分片
    ↓
entities/
├── people/           # 人员实体
├── organizations/    # 组织实体
├── technologies/     # 技术实体
├── products/         # 产品实体
├── systems/          # 系统实体
└── events/           # 事件实体
```

#### 索引分区

```
当索引文件超过 1000 行时：

entity-index.md → 触发分区
    ↓
entity-index/
├── overview.md      # 总览
├── A-M.md           # A-M 的实体
├── N-Z.md           # N-Z 的实体
└── by-type.md       # 按类型分类
```

```
    ├── entities/
    ├── concepts/
```

## 5. 技术选型

### 5.1 当前技术栈

#### 核心组件

| 组件 | 技术 | 理由 |
|------|------|------|
| 文档格式 | Markdown | 通用、易读、易写 |
| 知识展示 | Obsidian | 免费、本地、图谱视图 |
| LLM | [当前使用的 LLM] | 智能处理 |
| 版本控制 | Git（可选） | 版本追踪、协作 |

#### 目录结构

```
基于文件系统（非数据库）

优点:
- 简单直观
- 无需额外工具
- 易于备份和迁移
- 版本控制友好

缺点:
- 查询性能有限
- 复杂关系难表达
- 并发控制弱
```

### 5.2 未来技术选项

#### 数据库选项

当文档数量 > 10,000 时考虑：

```yaml
选项 1: SQLite
  适用场景：单用户或小团队
  优点：简单、嵌入式、无需服务器
  缺点：并发能力有限

选项 2: PostgreSQL
  适用场景：中大型团队
  优点：强大、可扩展、支持复杂查询
  缺点：需要数据库服务器

选项 3: Graph Database (Neo4j)
  适用场景：复杂关系网络
  优点：天然表达关系、图查询强大
  缺点：学习曲线、运维成本
```

#### 搜索引擎

当查询性能成为瓶颈时考虑：

```yaml
选项 1: Elasticsearch
  适用场景：全文搜索、复杂查询
  优点：强大、可扩展、生态成熟
  缺点：资源消耗大

选项 2: Meilisearch
  适用场景：中小型项目、快速部署
  优点：轻量、快速、易用
  缺点：功能相对简单

选项 3: Algolia
  适用场景：SaaS 偏好、无需自运维
  优点：托管服务、性能优秀
  缺点：成本较高
```

### 5.3 技术债务

#### 当前限制

1. **文件系统限制**
   - 文件数量上限：~10,000
   - 单文件大小上限：~10MB
   - 路径长度限制：260 字符（Windows）

2. **LLM 限制**
   - 上下文窗口限制
   - 推理成本
   - 响应时间

3. **并发限制**
   - 无锁机制
   - 无事务支持
   - 无冲突检测

#### 改进计划

```
短期（1-3 个月）:
- [ ] 实现简单的文件锁
- [ ] 添加备份机制
- [ ] 优化大文件处理

中期（3-6 个月）:
- [ ] 引入 SQLite 存储索引
- [ ] 实现查询缓存
- [ ] 添加批量处理

长期（6-12 个月）:
- [ ] 评估图数据库
- [ ] 实现 API 层
- [ ] 支持分布式部署
```

**日期**: 2026-04-17

**背景**:
- 需要存储和管理项目知识
- 考虑文件系统和数据库两种方案

**决策**:
- 使用文件系统存储所有文档
- 使用 Markdown 格式
- 暂不引入数据库

**理由**:
- 简单直观，学习成本低
- 无需额外工具，开箱即用
- 易于备份和迁移
- 版本控制友好
- 适合当前规模（< 1000 篇文档）

**后果**:
- ✅ 优点：简单、快速启动
- ⚠️ 缺点：查询性能有限，复杂关系难表达
- 📋 未来：当文档 > 10,000 时考虑引入数据库

### ADR-002: LLM 自动维护 Wiki

**状态**: 已采纳  
**日期**: 2026-04-17

**背景**:
- Wiki 需要持续更新和维护
- 人工维护成本高

**决策**:
- LLM 负责 Wiki 的创建、更新、维护
- 人类负责审查和决策

**理由**:
- 减少人工维护成本
- 提高更新频率
- 保持一致性
- 自动建立交叉引用

**后果**:
- ✅ 优点：自动化、高效
- ⚠️ 缺点：需要审查机制防止错误传播
- 📋 未来：建立质量检查和审查流程
