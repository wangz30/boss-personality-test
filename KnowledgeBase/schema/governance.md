## 1. 治理原则

### 1.1 核心原则

#### 单一事实来源 (Single Source of Truth)

- ✅ **原始文档** 是不可变的事实来源
- ✅ **Wiki 页面** 是从原始文档衍生的知识
- ✅ **如有冲突**，以原始文档为准

#### 责任分离 (Separation of Concerns)

- 👤 **人类** 负责：
  - 提供高质量原始文档
  - 审查 LLM 提取的信息
  - 做出决策和判断
  
- 🤖 **LLM** 负责：
  - 读取和解析文档
  - 提取和整合信息
  - 维护和更新 Wiki
  - 建立交叉引用

#### 持续演进 (Continuous Evolution)

- ✅ Wiki 是**活文档**，持续更新
- ✅ 定期审查和重构
- ✅ 基于反馈持续改进

#### 透明可追溯 (Transparency & Traceability)

- ✅ 所有变更有记录
- ✅ 所有信息有来源
- ✅ 所有决策有依据

## 2. 角色与职责

### 2.1 角色定义

#### 📝 文档作者 (Document Author)

**职责**:
- 创建高质量原始文档
- 确保文档结构清晰
- 提供准确的元数据
- 及时更新过时文档

**权限**:
- 创建/编辑 `raw-docs/` 中的文档
- 查看 Wiki 页面
- 建议 Wiki 修改

#### 🔍 审查员 (Reviewer)

**职责**:
- 审查 LLM 提取的信息
- 确认实体和概念定义
- 识别知识缺口
- 标记不准确内容

**权限**:
- 查看所有 Wiki 页面
- 标记需要审查的内容
- 请求 LLM 重新处理

#### 🏗️ 架构师 (Architect)

**职责**:
- 设计 Wiki 架构
- 定义分类体系
- 制定命名规范
- 审查重大变更

**权限**:
- 修改 Schema 配置
- 创建/修改模板
- 批准架构变更

#### 👨‍💼 管理员 (Admin)

**职责**:
- 管理用户权限
- 审查删除操作
- 处理冲突
- 审计合规性

**权限**:
- 所有权限
- 删除页面
- 修改权限设置

#### 🤖 LLM Agent

**职责**:
- 自动处理文档
- 维护 Wiki 页面
- 建立交叉引用
- 记录变更日志
- 识别知识缺口
- 回答查询

**权限**:
- 读取 `raw-docs/`
- 写入 `wiki/`
- 写入 `indexes/`
- 写入 `logs/`
- **不能** 删除原始文档
- **不能** 修改 Schema 配置（需人工批准）

## 3. 质量标准

### 3.1 文档质量标准

#### 原始文档

| 维度 | 标准 | 检查方式 |
|------|------|---------|
| 完整性 | 包含所有必要信息 | 人工审查 |
| 准确性 | 信息真实准确 | 人工审查 + LLM 验证 |
| 结构性 | 有清晰的章节结构 | LLM 自动检查 |
| 一致性 | 术语和格式一致 | LLM 自动检查 |
| 时效性 | 定期更新（≤ 90 天） | LLM 自动检查 |

#### Wiki 页面

| 维度 | 标准 | 检查方式 |
|------|------|---------|
| 元数据 | 完整且准确 | LLM 自动检查 |
| 摘要 | 准确反映原文 | 人工抽样审查 |
| 关联 | 至少 3 个交叉引用 | LLM 自动检查 |
| 链接 | 有效率 > 99% | LLM 自动检查 |
| 更新 | 随源文档同步更新 | LLM 自动触发 |

### 3.2 质量评分

#### 文档评分卡

```markdown
## 质量评分：[文档名称]

**总分**: ⭐⭐⭐⭐⭐ (5/5)

### 维度评分

- **完整性**: ⭐⭐⭐⭐⭐ (5/5)
  - 包含所有必要章节
  - 元数据完整
  
- **准确性**: ⭐⭐⭐⭐⭐ (5/5)
  - 信息经核实
  - 无矛盾内容
  
- **结构性**: ⭐⭐⭐⭐⭐ (5/5)
  - 章节清晰
  - 逻辑连贯
  
- **时效性**: ⭐⭐⭐⭐⭐ (5/5)
  - 最近更新（< 30 天）
  - 反映最新决策
```

#### Wiki 健康度

```markdown
## Wiki 健康度报告

**日期**: YYYY-MM-DD  
**总体评分**: ⭐⭐⭐⭐⭐ (5/5)

### 指标

- **覆盖率**: 85% ✅
  - 目标：80%
  - 实际：85%
  
- **关联度**: 92% ✅
  - 目标：90%
  - 实际：92%
  
- **时效性**: 78% ⚠️
  - 目标：80%
  - 实际：78%
  - 改进：需要更新 5 个过时页面
  
- **查询满意度**: 95% ✅
  - 目标：90%
  - 实际：95%
```

## 4. 审查机制

### 4.1 自动审查

LLM 自动执行以下审查：

#### 日常审查

- ✅ 元数据完整性
- ✅ 链接有效性
- ✅ 版本一致性
- ✅ 标签规范性

#### 定期审查（每周）

- ✅ 孤立页面识别
- ✅ 过时页面识别
- ✅ 知识缺口分析
- ✅ 质量评分更新

### 4.2 人工审查

#### 触发条件

以下情况需要人工审查：

- ⚠️ 删除实体或概念页面
- ⚠️ 标记文档为 deprecated
- ⚠️ 修改核心架构决策
- ⚠️ 创建新的综合论述
- ⚠️ 解决知识冲突

#### 审查流程

```
1. LLM 标记需要审查的内容
2. 发送通知给审查员
3. 审查员审查并提出意见
4. LLM 根据意见调整
5. 审查员批准
6. 执行变更
```

#### 审查清单

```markdown
## 审查清单

**审查对象**: [页面名称]  
**审查类型**: [创建/更新/删除/修改]  
**审查员**: [姓名]  
**日期**: YYYY-MM-DD

### 审查项

- [ ] 信息准确性
- [ ] 元数据完整性
- [ ] 交叉引用充分性
- [ ] 链接有效性
- [ ] 格式规范性

### 意见

[审查员填写]

### 决定

- [ ] ✅ 批准
- [ ] ⚠️ 需要修改
- [ ] ❌ 拒绝

**签名**: __________  
**日期**: __________
```

## 5. 冲突解决

### 5.1 冲突类型

#### 信息冲突

**场景**: 不同文档中的信息矛盾

**解决流程**:

```
1. 识别冲突（LLM 自动检测）
2. 标注冲突（使用⚠️标记）
3. 创建综合论述讨论冲突
4. 提交团队审查
5. 做出决策
6. 更新 Wiki 反映决策
```

#### 权限冲突

**场景**: 多个用户同时修改同一页面

**解决流程**:

```
1. LLM 检测并发修改
2. 保留所有版本
3. 通知相关用户
4. 协商解决方案
5. 合并或选择版本
6. 记录决策
```

#### 架构冲突

**场景**: 对 Wiki 架构有不同意见

**解决流程**:

```
1. 提出架构变更建议
2. 创建 ADR（架构决策记录）
3. 团队讨论
4. 投票或负责人决策
5. 实施变更
6. 更新 Schema 配置
```

### 5.2 升级机制

当无法在团队内解决时：

```
级别 1: 团队内部讨论
    ↓ (无法解决)
级别 2: 项目负责人决策
    ↓ (无法解决)
级别 3: 技术委员会审查
    ↓ (无法解决)
级别 4: CTO/技术总监决策
```

## 6. 合规要求

### 6.1 数据安全

#### 敏感信息处理

- 🔒 **禁止** 在 Wiki 中存储：
  - 密码和密钥
  - 个人身份信息（PII）
  - 商业机密
  - 客户数据

- ✅ **允许** 存储：
  - 技术架构
  - 设计决策
  - 会议纪要
  - 项目文档

#### 访问控制

| 信息类型 | 可见性 | 审批要求 |
|---------|--------|---------|
| 公开技术文档 | public | 无需审批 |
| 内部设计文档 | team | 团队成员 |
| 敏感决策 | private | 管理员审批 |

### 6.2 版权合规

#### 外部内容

- ✅ **允许**:
  - 引用公开文档（注明出处）
  - 使用开源协议内容
  - 合理使用（fair use）

- ❌ **禁止**:
  - 复制受版权保护内容
  - 侵犯知识产权
  - 未授权传播

#### 引用规范

```markdown
## 参考资料

### 外部资源

- [文章标题](URL) - [作者] - [来源] - [许可协议]
- [官方文档](URL) - [组织] - [版本]
```

### 6.3 审计追踪

#### 必须记录的操作

- ✅ 所有 Wiki 页面变更
- ✅ 权限变更
- ✅ 删除操作
- ✅ 架构变更
- ✅ 审查决策

#### 记录保留

| 记录类型 | 保留期限 | 存储位置 |
|---------|---------|---------|
| 变更日志 | 永久 | logs/change-log.md |
| 审查记录 | 2 年 | logs/review-log.md |
| 审计日志 | 永久 | 外部审计系统 |

## 7. 审计与报告

### 7.1 定期审计

#### 月度审计

**执行者**: 管理员  
**范围**: 全量检查

**审计内容**:

```markdown
## 月度审计报告

**月份**: YYYY-MM  
**审计员**: [姓名]

### 变更统计

- 新增页面：[数量]
- 更新页面：[数量]
- 删除页面：[数量]
- 审查操作：[数量]

### 合规检查

- [ ] 无敏感信息泄露
- [ ] 所有删除已审批
- [ ] 所有变更有记录
- [ ] 权限设置正确

### 质量问题

- [问题 1] - [严重性] - [状态]
- [问题 2] - [严重性] - [状态]

### 改进建议

- [建议 1]
- [建议 2]

### 签字

**审计员**: __________  
**日期**: __________
```

#### 季度审查

**执行者**: 架构师 + 管理员  
**范围**: 架构和流程

**审查内容**:

- Wiki 架构是否合理
- 工作流程是否高效
- 质量标准是否达成
- 工具和技术是否适用

### 7.2 临时审计

#### 触发条件

- 发现安全漏洞
- 重大质量事故
- 用户投诉
- 合规要求变更

#### 审计流程

```
1. 收到审计请求
2. 成立审计小组
3. 收集相关证据
4. 分析问题原因
5. 制定整改措施
6. 实施整改
7. 跟踪验证
```

### 7.3 报告机制

#### 报告类型

| 报告类型 | 频率 | 受众 | 内容 |
|---------|------|------|------|
| 周报 | 每周 | 团队成员 | 变更统计、质量问题 |
| 月报 | 每月 | 项目负责人 | 健康度、合规性 |
| 季报 | 每季 | 管理层 | 趋势分析、改进计划 |
| 年报 | 每年 | 所有利益相关者 | 年度总结、战略规划 |
