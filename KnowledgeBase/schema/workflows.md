## 1. 文档摄入流程

### 触发条件

- 用户添加新文档到 `raw-docs/` 目录
- 用户明确指示："请处理这个文档"

### 流程步骤

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 检测新文档                                          │
├─────────────────────────────────────────────────────────────┤
│ - 扫描 raw-docs/ 目录                                       │
│ - 识别未处理的文档（无对应 Summary）                        │
│ - 记录到摄入日志                                            │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 读取文档                                            │
├─────────────────────────────────────────────────────────────┤
│ - 读取文档内容                                              │
│ - 解析元数据（标题、作者、日期、类型）                      │
│ - 识别文档结构（章节、列表、代码块）                        │
│ - 验证文档质量（格式、完整性）                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 信息提取                                            │
├─────────────────────────────────────────────────────────────┤
│ - 提取核心摘要（100-200 字）                                  │
│ - 识别关键决策和结论                                        │
│ - 提取实体（人、组织、技术、产品等）                        │
│ - 提取概念（模式、原则、方法等）                            │
│ - 识别待办事项和行动项                                      │
│ - 标注优先级和状态                                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 4: 创建摘要页面                                        │
├─────────────────────────────────────────────────────────────┤
│ - 使用 summary-template.md                                  │
│ - 填写元数据（type, created, tags, source）                 │
│ - 编写核心摘要                                              │
│ - 列出关键要点                                              │
│ - 建立交叉引用                                              │
│ - 保存到 wiki/summaries/                                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: 更新/创建实体页面                                   │
├─────────────────────────────────────────────────────────────┤
│ - 检查实体是否已存在                                        │
│ - 如不存在，使用 entity-template.md 创建                     │
│ - 如存在，更新相关信息                                      │
│ - 添加新的交叉引用                                          │
│ - 更新版本号和更新时间                                      │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 6: 更新/创建概念页面                                   │
├─────────────────────────────────────────────────────────────┤
│ - 检查概念是否已存在                                        │
│ - 如不存在，使用 concept-template.md 创建                    │
│ - 如存在，更新相关信息                                      │
│ - 添加新的交叉引用                                          │
│ - 更新版本号和更新时间                                      │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 7: 更新索引                                            │
├─────────────────────────────────────────────────────────────┤
│ - 更新 entity-index.md                                      │
│ - 更新 concept-index.md                                     │
│ - 更新 document-index.md                                    │
│ - 更新 tag-index.md                                         │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 8: 记录日志                                            │
├─────────────────────────────────────────────────────────────┤
│ - 记录到 ingestion-log.md                                   │
│ - 记录到 change-log.md                                      │
│ - 更新统计信息                                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 9: 用户确认                                            │
├─────────────────────────────────────────────────────────────┤
│ - 向用户展示提取的关键信息                                  │
│ - 询问是否有遗漏或错误                                      │
│ - 根据反馈进行调整                                          │
└─────────────────────────────────────────────────────────────┘
```

### 输出产物

- ✅ 摘要页面（wiki/summaries/）
- ✅ 实体页面（wiki/entities/）
- ✅ 概念页面（wiki/concepts/）
- ✅ 索引更新（KnowledgeBase/indexes/）
- ✅ 日志记录（KnowledgeBase/logs/）

### 质量标准

- [ ] 摘要准确反映原文档核心内容
- [ ] 实体识别完整（无遗漏）
- [ ] 概念定义清晰
- [ ] 交叉引用正确
- [ ] 元数据完整
- [ ] 日志记录详细

## 2. 查询响应流程

### 触发条件

- 用户提出问题
- 用户请求信息检索

### 流程步骤

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 理解查询                                            │
├─────────────────────────────────────────────────────────────┤
│ - 分析查询意图                                              │
│ - 识别查询类型（知识/导航/综合/维护）                       │
│ - 提取关键词和实体                                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 检索 Wiki                                           │
├─────────────────────────────────────────────────────────────┤
│ - 优先检索 synthesis/（综合论述）                           │
│ - 检索 entities/（实体页面）                                │
│ - 检索 concepts/（概念页面）                                │
│ - 记录引用来源                                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 评估结果                                            │
├─────────────────────────────────────────────────────────────┤
│ - 信息是否充分？                                            │
│   - 是 → 进入 Step 5                                        │
│   - 否 → 进入 Step 4                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 4: 检索原始文档                                        │
├─────────────────────────────────────────────────────────────┤
│ - 检索 summaries/（文档摘要）                               │
│ - 必要时检索 raw-docs/（原始文档）                          │
│ - 识别知识缺口                                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: 综合回答                                            │
├─────────────────────────────────────────────────────────────┤
│ - 组织信息结构                                              │
│ - 引用 Wiki 页面（使用链接）                                 │
│ - 标注置信度                                                │
│ - 指出知识缺口（如有）                                      │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 6: 记录查询                                            │
├─────────────────────────────────────────────────────────────┤
│ - 记录到 query-log.md                                       │
│ - 标注满意度（如用户反馈）                                  │
│ - 识别优化机会                                              │
└─────────────────────────────────────────────────────────────┘
```

### 响应模板

```markdown
## 回答

[直接回答用户问题]

### 相关页面

- [Entity-名称](entities/Entity-名称.md) - [相关说明]
- [Concept-名称](concepts/Concept-名称.md) - [相关说明]
- [Summary-文档](summaries/Summary-文档.md) - [相关说明]

### 知识缺口

⚠️ **注意**: 关于 [方面]，当前 Wiki 中信息不足。
建议：[如何填补]

### 置信度

- [主张 1]: 90% - 基于 [来源]
- [主张 2]: 70% - 基于 [来源]
```

## 3. Wiki 维护流程

### 触发条件

- 定期执行（每周）
- 添加新文档后
- 用户请求维护

### 流程步骤

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 检查实体页面                                        │
├─────────────────────────────────────────────────────────────┤
│ - 识别未链接到任何文档的实体（孤立实体）                    │
│ - 识别超过 30 天未更新的实体                                 │
│ - 识别被频繁引用但未创建的实体                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 检查概念页面                                        │
├─────────────────────────────────────────────────────────────┤
│ - 识别未链接到任何文档的概念（孤立概念）                    │
│ - 识别超过 30 天未更新的概念                                 │
│ - 识别被频繁引用但未创建的概念                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 检查交叉引用                                        │
├─────────────────────────────────────────────────────────────┤
│ - 验证所有链接是否有效                                      │
│ - 识别断开的链接                                            │
│ - 修复或标记无效链接                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 4: 识别综合机会                                        │
├─────────────────────────────────────────────────────────────┤
│ - 识别有 5+ 个相关文档的主题                                 │
│ - 建议创建 synthesis 页面                                    │
│ - 列出关键洞察和趋势                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: 识别知识缺口                                        │
├─────────────────────────────────────────────────────────────┤
│ - 分析查询日志中的未满足查询                                │
│ - 识别频繁提及但未记录的内容                                │
│ - 建议填补措施                                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 6: 生成维护报告                                        │
├─────────────────────────────────────────────────────────────┤
│ - 总结发现的问题                                            │
│ - 列出建议的行动                                            │
│ - 提交用户审查                                              │
└─────────────────────────────────────────────────────────────┘
```

### 维护报告模板

```markdown
## Wiki 维护报告

**日期**: YYYY-MM-DD  
**范围**: 全量检查

### 发现的问题

1. **孤立实体**: [数量] 个
   - [Entity-名称 1] - 30 天未更新
   - [Entity-名称 2] - 无交叉引用

2. **断开的链接**: [数量] 个
   - [页面 1] → [目标页面] - 目标不存在

3. **过时信息**: [数量] 个
   - [页面 1] - 最后更新：60 天前

### 建议的行动

1. [ ] 更新 [Entity-名称] - 优先级：高
2. [ ] 修复 [断开的链接] - 优先级：中
3. [ ] 创建 [Synthesis-主题] - 优先级：高

### 知识缺口

- [缺口 1] - 影响：[描述]
- [缺口 2] - 影响：[描述]

### 统计

- 实体页面：[数量]
- 概念页面：[数量]
- 摘要页面：[数量]
- 综合论述：[数量]
```

## 4. 质量检查流程

### 触发条件

- 每次 Wiki 更新后
- 定期执行（每周）
- 用户请求检查

### 检查清单

#### 元数据检查

```markdown
- [ ] type 字段正确（entity/concept/summary/synthesis）
- [ ] created 字段存在且格式正确
- [ ] updated 字段最新
- [ ] version 字段符合语义化版本
- [ ] tags 字段使用预定义标签
- [ ] related 字段链接有效
- [ ] source 字段指向原始文档
- [ ] status 字段正确（draft/review/stable/deprecated）
- [ ] visibility 字段正确（public/team/private）
```

#### 内容质量检查

```markdown
- [ ] 标题清晰准确
- [ ] 概述简洁（2-3 句话）
- [ ] 内容结构化（使用标题、列表）
- [ ] 交叉引用充分（至少 3 个）
- [ ] 链接有效（无 404）
- [ ] 无拼写和语法错误
- [ ] 术语一致
- [ ] 信息准确（与源文档一致）
```

#### 关联关系检查

```markdown
- [ ] 实体页面有关联的概念
- [ ] 概念页面有关联的实体
- [ ] 摘要页面链接到原始文档
- [ ] 综合论述引用足够的摘要
- [ ] 索引页面包含所有页面
```

### 自动修复

LLM 会自动修复以下问题：

- ✅ 缺失的元数据
- ✅ 过时的版本号
- ✅ 格式不一致
- ✅ 断开的链接（标记）

### 需要人工审查

以下问题需要人工审查：

- ⚠️ 内容冲突
- ⚠️ 信息不准确
- ⚠️ 删除页面
- ⚠️ 重大架构变更

## 5. 版本发布流程

### 触发条件

- 累积足够多的变更
- 重大架构更新
- 用户请求发布

### 版本编号

遵循语义化版本：

```
MAJOR.MINOR.PATCH
  │     │     │
  │     │     └─ 向后兼容的错误修复
  │     └─────── 向后兼容的新功能
  └───────────── 不向后兼容的重大变更
```

### 发布步骤

```
1. 审查变更日志
2. 确定版本号
3. 更新所有页面的 version 字段
4. 创建发布说明
5. 标记 Git tag（如使用 Git）
6. 通知团队成员
```

### 发布说明模板

```markdown
# Release v[版本号]

**发布日期**: YYYY-MM-DD  
**类型**: Major/Minor/Patch

## 新增功能

- [功能 1] - [描述]
- [功能 2] - [描述]

## 改进

- [改进 1] - [描述]
- [改进 2] - [描述]

## 修复

- [Bug 1] - [描述]
- [Bug 2] - [描述]

## 重大变更

⚠️ **注意**: [描述不向后兼容的变更]

## 统计

- 新增页面：[数量]
- 更新页面：[数量]
- 删除页面：[数量]

## 升级指南

[如何升级到此版本]
```

## 6. 异常处理流程

### 6.1 文档处理失败

**症状**: 文档无法正确解析

**处理步骤**:

```
1. 检查文档格式（是否为 Markdown）
2. 检查文件编码（应为 UTF-8）
3. 检查文件完整性
4. 尝试手动提取关键信息
5. 记录到 ingestion-log.md（错误部分）
6. 通知用户
```

### 6.2 Wiki 页面损坏

**症状**: 页面内容异常或无法读取

**处理步骤**:

```
1. 从备份恢复（如有）
2. 检查变更日志找到最后正常版本
3. 回滚到正常版本
4. 记录到 change-log.md
5. 分析损坏原因
6. 实施预防措施
```

### 6.3 索引不一致

**症状**: 索引与实际页面不匹配

**处理步骤**:

```
1. 扫描所有页面
2. 重建索引
3. 验证链接
4. 记录到 change-log.md
5. 分析不一致原因
```

### 6.4 知识冲突

**症状**: 不同文档中的信息矛盾

**处理步骤**:

```
1. 识别冲突来源
2. 标注冲突（使用⚠️标记）
3. 创建综合论述讨论冲突
4. 提出解决方案建议
5. 提交团队审查
6. 更新 Wiki 反映决策
```
